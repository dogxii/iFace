什么是环境变量？

💡 核心概念

环境变量是存储在操作系统或部署平台中的键值对配置，用于存放敏感信息（如 API Key、Secret）和环境相关的配置（如域名、端口号）。

在 Node.js / Vite 项目中：

  • .env 文件用于本地开发时的配置
  • .env.example 是模板文件，告诉协作者需要配置哪些变量（不含真实值）
  • 部署时在平台（如 Vercel）的设置面板中填写真实值

🔒 为什么不能把密钥写在代码里？

  • 代码会推送到 GitHub，所有人可见
  • API Key 泄露会导致账号被盗用、产生费用
  • 环境变量只在运行时注入，不会被打包到前端代码中（VITE_ 前缀的除外）
⚠️
Vite 特殊规则：只有以 VITE_ 为前缀的环境变量才会暴露给前端浏览器代码。其他变量只在服务端（如 Vercel Serverless 函数）可用。这是 Vite 的安全设计。
🔍

逐行解析 .env.example

📄 完整文件内容

.env.example # ─── GitHub OAuth ───────────────────────────────────────────────────────────── # Create a GitHub OAuth App at: https://github.com/settings/developers # Homepage URL: https://your-app.vercel.app # Callback URL: https://your-app.vercel.app/api/auth # # Then set these in Vercel project settings → Environment Variables: # GITHUB_CLIENT_ID (server-side, used in api/auth.js) # GITHUB_CLIENT_SECRET (server-side, used in api/auth.js) # APP_ORIGIN (server-side, e.g. https://your-app.vercel.app) # # And this in your local .env (prefix VITE_ so Vite exposes it to the browser): VITE_GITHUB_CLIENT_ID=your_github_oauth_app_client_id # ─── 1.0 External Smoke Checks ──────────────────────────────────────────────── # These are CLI-only variables used by `bun run check:external`. # Do not prefix them with VITE_; they should not be exposed to the browser. # IFACE_AI_API_KEY=your_ai_api_key IFACE_AI_BASE_URL=https://api.example.com/v1 IFACE_AI_MODEL=your-model-id IFACE_AI_TIMEOUT_MS=60000 IFACE_GIST_TOKEN=your_gist_token

第一部分：GitHub OAuth 配置

🔑 涉及的 4 个变量

变量名 存放位置 用途
VITE_GITHUB_CLIENT_ID 本地 .env 前端构建时注入，用于拼接 OAuth 授权 URL
GITHUB_CLIENT_ID Vercel 环境变量 服务端 api/auth.js 用于 code 换 token
GITHUB_CLIENT_SECRET Vercel 环境变量 服务端 api/auth.js 用于 code 换 token（绝不能暴露给前端）
APP_ORIGIN Vercel 环境变量 OAuth 回调后重定向的域名（CORS 白名单）

⚡ OAuth 完整流程图

① 用户点击登录
→
② 跳转 GitHub 授权页
→
③ 用户同意授权
→
④ 回调到 /api/auth
→
⑤ 服务端换 Token
→
⑥ Token 返回前端

📐 为什么 Client ID 出现了两次？

同一个 Client ID 在两个地方使用，但目的不同：

  • 前端（VITE_GITHUB_CLIENT_ID）：拼接授权 URL，让用户跳转到 GitHub 登录页
  • 后端（GITHUB_CLIENT_ID）：和 Client Secret 一起，用授权码换取 Access Token

前端的 VITE_ 前缀让 Vite 在构建时把它注入到浏览器代码中；后端的变量没有 VITE_ 前缀，只在 Vercel Serverless 函数中可用。

🚨
安全警告：GITHUB_CLIENT_SECRET 绝对不能加 VITE_ 前缀！否则它会被打包到前端 JS 中，任何人都能看到你的 Secret，导致你的 OAuth App 被滥用。
⚙️

Vite 环境变量机制详解

🏷️ VITE_ 前缀的作用

TypeScript // ✅ 以 VITE_ 开头 → 浏览器端可用 import.meta.env.VITE_GITHUB_CLIENT_ID // → 编译后会被替换为实际值: "Ov23liXXXXXXXXXX" // ❌ 没有 VITE_ 前缀 → 浏览器端为 undefined import.meta.env.GITHUB_CLIENT_SECRET // → 编译后为 undefined，保护了敏感信息 // ✅ 在 api/auth.js (Node.js) 中可以访问所有变量 process.env.GITHUB_CLIENT_SECRET // → 正常读取，因为这是服务端代码

📁 变量查找优先级

Vite 按以下优先级加载环境变量（数字越大优先级越高）：

优先级 文件 说明
1（最低） .env 所有模式通用
2 .env.local 本地覆盖，.gitignore 忽略
3 .env.[mode] 如 .env.production
4（最高） .env.[mode].local 本地模式覆盖
💡
最佳实践：把 .env.example 提交到 Git 作为模板；.env.local 放真实密钥并加入 .gitignore；部署时在 Vercel 面板设置环境变量。

第二部分：外部服务冒烟测试

📋 变量说明

变量名 用途 必需？
IFACE_AI_API_KEY AI 服务的 API Key，用于 bun run check:external -- --ai AI 冒烟测试时必需
IFACE_AI_BASE_URL AI 服务的 API 地址 可选，默认使用应用内配置
IFACE_AI_MODEL 测试用的模型 ID 可选，默认使用应用内配置
IFACE_AI_TIMEOUT_MS AI 请求超时时间（毫秒） 可选，默认 60000
IFACE_GIST_TOKEN GitHub Personal Access Token，用于 Gist 同步冒烟测试 Gist 冒烟测试时必需

🚀 使用场景

这些变量不是给前端用的，而是给 CI/CD 脚本用的，用于在发布前验证外部服务是否正常：

Shell # 测试 AI 服务是否可用 bun run check:external -- --ai # 测试 Gist 同步是否可用 bun run check:external -- --gist # 测试并记录结果到文件 bun run smoke:external:ai bun run smoke:external:gist
📌
注意前缀：这些变量都没有 VITE_ 前缀，所以不会暴露给浏览器。它们只在 Node.js 脚本（scripts/checkExternalServices.ts）中通过 process.env 读取。这是正确的安全做法——API Key 只在服务端/CLI 中使用。

🔄 IFACE_GIST_TOKEN vs GitHub OAuth Token

这两个是不同的 Token：

对比 IFACE_GIST_TOKEN GitHub OAuth Token
获取方式 在 GitHub Settings → Developer settings → Personal access tokens 手动创建 通过 OAuth 授权流程自动获取
使用场景 CLI 冒烟测试脚本 用户在浏览器中同步数据
存储位置 本地 shell 环境变量 浏览器 localStorage
权限范围 可以设置 gist + 其他权限 只有 gist 权限

实战：如何配置

本地开发配置步骤

  1. 创建 .env.local 文件

    在项目根目录创建 .env.local（已在 .gitignore 中，不会被提交）

    # .env.local VITE_GITHUB_CLIENT_ID=Ov23liXXXXXXXXXX

  2. 创建 GitHub OAuth App

    打开 github.com/settings/developers，点击 New OAuth App：

    Homepage URL http://localhost:5173（本地开发用 localhost）
    Callback URL http://localhost:5173/api/auth

    创建后复制 Client ID 填入 .env.local。

  3. 启动开发服务器

    Shell bun install bun run dev

    Vite 会自动加载 .env.local，VITE_GITHUB_CLIENT_ID 就可以在前端代码中通过 import.meta.env.VITE_GITHUB_CLIENT_ID 访问了。

☁️ Vercel 部署配置步骤

  1. 在 Vercel 项目设置中添加环境变量

    进入 Vercel Dashboard → 你的项目 → Settings → Environment Variables，添加：

    Vercel Environment Variables VITE_GITHUB_CLIENT_ID = Ov23liXXXXXXXXXX # 所有环境 GITHUB_CLIENT_ID = Ov23liXXXXXXXXXX # 所有环境 GITHUB_CLIENT_SECRET = secret_xxxxxxxxxxxx # 所有环境（保密！） APP_ORIGIN = https://iface.vercel.app # Production

  2. 更新 GitHub OAuth App 的 Callback URL

    回到 GitHub OAuth App 设置，把 Callback URL 改为生产域名：

    Authorization callback URL: https://iface.vercel.app/api/auth

  3. 重新部署

    环境变量修改后需要重新部署才能生效。Vercel 会自动在 git push 时触发部署。

✅
配置完成后：用户在应用中点击「用 GitHub 登录」→ 跳转 GitHub 授权 → 同意后自动回调 → Token 存入 localStorage → 即可使用 Gist 云同步功能。

常见问题排查

❌ 登录后报错 "VITE_GITHUB_CLIENT_ID is not set"

原因：.env.local 文件不存在或变量名拼写错误。

解决：确认项目根目录有 .env.local 文件，且变量名完全匹配（区分大小写）。

❌ GitHub 授权后回调到空白页

原因：OAuth App 的 Callback URL 与实际部署地址不匹配。

解决：检查 GitHub OAuth App 设置中的 Authorization callback URL 是否正确。

❌ "GitHub API 401: Bad credentials"

原因：Token 过期或被撤销。

解决：在应用设置中退出登录，重新走 OAuth 授权流程获取新 Token。

❌ Gist 同步后数据没有出现

原因：浏览器缓存了旧的 IndexedDB 数据。

解决：刷新页面，或在设置面板中手动点击「从云端拉取」。

安全设计总结

安全措施 实现方式
Client Secret 不泄露 只在 Vercel Serverless 函数中使用，没有 VITE_ 前缀
最小权限原则 OAuth 只申请 gist 一个 scope
CSRF 防护 OAuth 流程中使用 state 参数，回调时校验一致性
Token 不持久化到服务端 Token 通过 URL hash（#token=xxx）返回，不会出现在服务器日志
Token 存储在客户端 存入 localStorage，仅当前浏览器可用
Gist 使用私有模式 创建时 public: false，只有 Token 持有者能访问
不上传内置题库 只同步用户数据，内置题从本地 fetch，节省 95% 空间
