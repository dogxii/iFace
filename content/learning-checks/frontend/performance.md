# 性能优化 测试一下

## perf-001

### Q1 single | Core Web Vitals

当前 Core Web Vitals 的核心指标是哪一组？

- [x] A. LCP、INP、CLS。
- [ ] B. FCP、DNS、TCP。
- [ ] C. TTFB、HTTP、CSS。
- [ ] D. FPS、GPU、RAM。

**解释**：Core Web Vitals 聚焦加载体验、交互响应和视觉稳定性，当前核心指标是 LCP、INP、CLS。

### Q2 multiple | 指标含义

哪些指标解释正确？

- [x] A. LCP 衡量视口内最大内容元素完成渲染的时间。
- [x] B. INP 衡量交互到下一次绘制的响应延迟。
- [x] C. CLS 衡量非预期布局偏移的累计分数。
- [ ] D. LCP 衡量所有接口请求总耗时。
- [ ] E. CLS 越大说明页面越稳定。

**解释**：LCP 看加载主内容，INP 看交互反馈，CLS 看布局稳定；CLS 分数越低越好。

### Q3 multiple | 测量方式

哪些说法正确？

- [x] A. 判断站点是否达标通常看真实用户数据的第 75 百分位。
- [x] B. Lighthouse 适合实验室复现和调试。
- [x] C. RUM 埋点能反映真实用户设备、网络和页面路径。
- [ ] D. 单次本地跑分可以代表所有用户体验。
- [ ] E. Core Web Vitals 只需要在开发环境测一次。

**解释**：性能指标要区分实验室数据和真实用户数据。真实体验受设备、网络和业务路径影响很大。

## perf-002

### Q1 single | Reflow

重排 Reflow 主要发生在什么情况下？

- [ ] A. 只改变文字颜色。
- [x] B. 元素几何信息变化，需要重新计算布局。
- [ ] C. 只改变透明度且在合成层上。
- [ ] D. 只更新 localStorage。

**解释**：宽高、位置、字体、display 等几何变化可能触发布局计算。

### Q2 multiple | 减少重排重绘

哪些做法有助于减少重排或重绘成本？

- [x] A. 批量读写 DOM，避免布局抖动。
- [x] B. 动画优先使用 `transform` 和 `opacity`。
- [x] C. 用 class 一次性切换多条样式。
- [x] D. 对独立区域使用 `contain` 限制影响范围。
- [ ] E. 每次循环里先写样式再读 `offsetHeight`。

**解释**：交替读写布局属性容易触发 forced reflow。应批量读取、批量写入。

### Q3 multiple | Paint 与 Composite

哪些说法正确？

- [x] A. 改 `background-color` 通常会触发 Paint。
- [x] B. 合成层上的 `transform` 变化通常可只走 Composite。
- [x] C. 大面积阴影、滤镜和模糊可能带来高 Paint 成本。
- [ ] D. Repaint 一定比 Composite 更快。
- [ ] E. 所有 CSS 属性变化成本都一样。

**解释**：不同属性触发的渲染阶段不同。优化时要看 Layout、Paint、Composite 的实际开销。

## perf-003

### Q1 single | 图片懒加载

图片懒加载的核心目标是什么？

- [x] A. 延后加载视口外图片，减少首屏资源竞争。
- [ ] B. 把所有图片同步加载完。
- [ ] C. 禁止浏览器缓存图片。
- [ ] D. 把图片都转成 JSON。

**解释**：懒加载让首屏关键资源优先，视口外图片接近可见时再加载。

### Q2 multiple | 实现方式

哪些方式可用于图片懒加载？

- [x] A. 原生 `loading="lazy"`。
- [x] B. IntersectionObserver。
- [x] C. 滚动监听加节流作为兜底。
- [ ] D. 每 1ms 遍历所有图片并读取布局。
- [ ] E. 删除所有 `src` 且永不恢复。

**解释**：现代优先原生 lazy 或 IntersectionObserver，滚动监听要作为兼容兜底并节流。

### Q3 multiple | 避免副作用

图片懒加载时还要注意什么？

- [x] A. 预留宽高或使用 `aspect-ratio`，减少 CLS。
- [x] B. 设置合适 rootMargin 提前加载，避免用户看到空白。
- [x] C. 加载后取消观察，减少后续开销。
- [ ] D. 懒加载 LCP 主图一定能优化 LCP。
- [ ] E. 所有图片都应该懒加载，包括首屏关键图。

**解释**：首屏 LCP 图片通常应优先加载，而不是懒加载。懒加载主要用于非首屏资源。

## perf-004

### Q1 single | 防抖节流选择

搜索框输入联想请求更适合使用什么？

- [x] A. 防抖。
- [ ] B. 节流。
- [ ] C. 长轮询。
- [ ] D. 递归渲染。

**解释**：输入停止一小段时间后再请求，能避免每个字符都触发接口。

### Q2 multiple | 节流适用场景

哪些场景更适合节流？

- [x] A. scroll 中周期性更新阅读进度。
- [x] B. mousemove 中限制计算频率。
- [x] C. resize 中持续采样布局变化。
- [ ] D. 用户停止输入后只提交最后一次搜索。
- [ ] E. 组件卸载时清理订阅。

**解释**：节流适合持续事件中稳定频率执行；防抖适合只关心最后一次。

### Q3 multiple | 性能边界

防抖节流优化时，哪些判断正确？

- [x] A. 它们减少函数触发频率，但不直接减少单次函数成本。
- [x] B. 需要在组件卸载时取消未执行的 timer。
- [x] C. 高频交互中仍要关注主线程任务是否过长。
- [ ] D. 加了防抖就不会有任何卡顿。
- [ ] E. 节流函数不需要保留最新参数。

**解释**：防抖节流是频率控制，不是性能银弹。单次任务过重仍会造成 INP 问题。

## perf-005

### Q1 single | 强缓存

强缓存命中时，浏览器会怎样？

- [x] A. 不发请求，直接使用本地缓存。
- [ ] B. 一定发送 `If-None-Match`。
- [ ] C. 一定返回 304。
- [ ] D. 重新下载完整资源。

**解释**：强缓存由 Cache-Control 或 Expires 判断，新鲜期内无需向服务器确认。

### Q2 multiple | 协商缓存

哪些说法正确？

- [x] A. ETag 配合 If-None-Match。
- [x] B. Last-Modified 配合 If-Modified-Since。
- [x] C. 资源未变时服务端可返回 304。
- [ ] D. 协商缓存命中时一定返回完整响应体。
- [ ] E. 协商缓存不需要网络请求。

**解释**：协商缓存需要一次网络确认，304 响应不返回完整资源体。

### Q3 multiple | 性能缓存策略

哪些策略更适合性能优化？

- [x] A. 带内容 hash 的 JS/CSS 使用长缓存。
- [x] B. HTML 入口谨慎长缓存，避免版本更新困难。
- [x] C. CDN 和浏览器缓存要区分设计。
- [x] D. 私有数据避免被公共缓存误缓存。
- [ ] E. 所有资源统一 `no-store` 性能最好。

**解释**：缓存要按资源更新频率和隐私属性设计。统一 no-store 会失去缓存收益。

## perf-012

- [x] C. 优化字体加载和 fallback 度量。
