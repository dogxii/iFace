# Prompt工程测试一下

## prompt-001

### Q1 single | Prompt工程定义

什么最准确描述 Prompt Engineering？

- [ ] A. 给模型加礼貌用语，让回答看起来更友好。
- [x] B. 围绕任务、上下文、约束和输出格式设计并迭代模型输入。
- [ ] C. 只适用于闲聊，不适用于结构化任务。
- [ ] D. 通过改 Prompt 直接修改模型权重。

**解释**：Prompt Engineering 是把通用模型能力转成稳定产品能力的输入协议设计，不是简单润色文案。

### Q2 multiple | 工程价值

Prompt Engineering 在 LLM 应用里通常带来哪些价值？

- [x] A. 更快验证业务想法
- [x] B. 降低直接训练或微调的成本
- [x] C. 提高格式、风格和行为边界的可控性
- [ ] D. 自动替代权限控制和事实校验
- [ ] E. 保证模型永远不幻觉

**解释**：Prompt 是控制层的一部分，能降低试错成本，但不能替代模型外的安全和校验机制。

### Q3 multiple | 适用边界

哪些问题通常不能只靠 Prompt 彻底解决？

- [x] A. 实时事实查询
- [x] B. 后端权限控制
- [x] C. 工具参数强校验
- [x] D. 高风险动作审计
- [ ] E. 明确输出字段顺序

**解释**：Prompt 可以约束模型行为，但实时数据、权限和执行边界需要系统能力兜底。

## prompt-002

### Q1 single | 角色分层

system prompt 和 user prompt 的主要区别是什么？

- [ ] A. user prompt 总是优先级最高。
- [x] B. system prompt 定义全局规则，user prompt 表达当前任务输入。
- [ ] C. system prompt 只能写输出格式。
- [ ] D. 二者不需要区分。

**解释**：分层设计能把稳定系统策略和动态用户需求分开管理。

### Q2 multiple | 分开设计

为什么要把 system prompt 和 user prompt 分开？

- [x] A. 系统规则更容易复用
- [x] B. 动态输入更容易替换
- [x] C. 异常输出更容易定位原因
- [x] D. 更容易降低提示注入风险
- [ ] E. 可以让用户覆盖安全规则

**解释**：系统层负责稳定边界，用户层负责当前任务；二者混在一起会难以治理。

### Q3 multiple | 冲突处理

当 user prompt 要求违反 system prompt 时，哪些处理更合理？

- [x] A. 遵循 system prompt 和外部安全策略
- [x] B. 必要时说明无法执行的原因
- [x] C. 不让用户输入覆盖高优先级规则
- [ ] D. 按用户要求忽略系统规则
- [ ] E. 随机选择一个要求执行

**解释**：冲突策略要明确，否则模型容易在冲突指令之间摇摆。
